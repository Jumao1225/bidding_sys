# 项目概述
**目标：** 开发一个利用 AI 技术分析招标文件，并辅助用户撰写投标书的 Web 应用系统。
**核心价值：** 帮助标书编写人员快速解析招标文件中冗长的要求，识别公司是否具备投标资格（匹配度分析），发现潜在风险项，进行初步的成本与报价测算，并自动生成投标书大纲草稿，大幅提升编写效率与竞标成功率。

# 用户画像
- **标书编写专员/售前工程师**：系统的主要使用者，负责阅读招标文件，整理公司资质，评估成本与风险，撰写并汇总最终的投标书。他们需要工具来快速定位核心要求、排查风险、测算底价并起草文档结构。

# 用户故事 (User Stories)
1. **作为标书编写专员**，我希望能够**上传 PDF 格式的招标文件**，以便系统能自动解析其中的文本内容。
2. **作为标书编写专员**，我希望系统能**自动提取招标文件的核心资质要求**，并与我公司已有的资质进行对比分析，分类为“完全符合”、“需努力改进”、“完全不符合”，以便我快速判断投标可行性。
3. **作为标书编写专员**，我希望系统能**自动识别出标书中的风险项**（如严苛的违约条款、不合理的付款条件等），以便我提前制定应对策略或决定放弃。
4. **作为标书编写专员**，我希望系统能根据标书中的采购清单**参考市场价格进行初步计算，估算出成本底价**，让我在竞价阶段对利润空间和报价底线有一个大体的认识。
5. **作为标书编写专员**，我希望能够**与招标文件进行交互式对话问答**，以便快速检索特定条款细节。
6. **作为标书编写专员**，我希望系统能**根据招标文件要求一键生成 Word 格式的投标书大纲/草稿**，以便我能在此基础上直接添加详细内容。
7. **作为系统管理员/高阶编写员**，我希望系统能够具备**可扩展的 AI 技能 (Skills)**（例如联网查询最新法规、对接内部 ERP 系统），以便 AI 能够结合外部知识和内部数据，给出更精准的风险判断。
8. **作为标书编写专员**，我希望系统能够**自动提取并醒目提示招标文件中的“废标项/一票否决项”**（如带 * 号的实质性条款），以防因遗漏导致直接废标。
9. **作为标书编写专员**，我希望系统能**解析“评标办法/打分表”**，并给出高分策略（例如重点突出哪些技术指标或提供几份业绩证明），指导我写出更容易拿高分的标书。
10. **作为标书编写专员**，我希望系统能根据招标文件末尾的**“投标文件格式”要求，严格生成大纲模板**，并在可能的情况下自动生成**技术/商务响应偏离表**草稿。

# 功能性需求
1. **文档管理模块**：
   - 支持上传 PDF 格式的招标文件。
   - 解析 PDF 文本内容（提取非结构化文本和结构化表格）。
2. **资质匹配与分析 (AI 核心)**：
   - 能够配置或导入“我公司的资质列表”。
   - AI 自动提取招标文件中的“资质要求”并与公司资质比对。
   - 输出清晰的分类评估结果（绿/黄/红 三级判定）。
3. **风险与废标项排查 (AI 核心)**：
   - 扫描全文，高亮并提取潜在的商务风险、法律风险、技术风险等。
   - **强制提取“废标项/一票否决项”**（如标有 * 号或“实质性响应”字样的条款），并要求用户逐一确认是否满足。
4. **评标标准与高分策略分析**：
   - 自动提取招标文件中的“评标办法/打分表”。
   - 解析各项评分比重（价格分、技术分、商务分），并提供针对性的高分策略建议（如提示需要准备特定的业绩合同或资质证明）。
5. **成本与报价辅助测算**：
   - AI 自动识别并提取招标文件中的采购物品/服务清单（BOM/BOQ）。
   - 提供一个基础的“市场价格参考库”（支持用户自行维护或导入）。
   - 系统自动对齐采购清单与参考价格，生成基础成本核算表，辅助用户制定最终竞价。
6. **智能问答 (Chat with PDF)**：
   - 提供基于当前文件的问答聊天窗口，准确定位并引用文档原文。
7. **标书大纲与偏离表生成**：
   - AI 精准定位并解析招标文件中的“投标文件格式/目录要求”，以此为依据生成**结构完全一致**的投标书大纲及模板。
   - 自动生成“技术/商务响应偏离表”草稿，辅助填入“完全响应”等默认值，并高亮需要人工核对的差异项。
   - 支持一键下载为 Word (.docx) 或 Excel (.xlsx) 格式文件。
8. **AI 技能插件系统 (Skill Mechanism)**：
   - 提供底层动态插拔的插件系统，支持注册自定义技能。
   - 核心代理 (Agent) 可利用大模型的 Function Calling/Tool Calling 能力，在阅读标书或回答问题时，自主判断并触发相应技能（如进行全网搜索、查询外部数据库库等）。

# 非功能性需求
1. **系统架构与模式**：
   - 基于 B/S 架构，通过现代主流浏览器访问。
   - **后端架构**：采用“UI 工具箱直达 + Supervisor 智能包工头”的混合路由架构，底层 Agent 封装为无状态 Tools。
   - **Agent 编排框架**：采用 **LangGraph** 进行复杂的多智能体工作流（State Graph）流转与调度。
2. **性能与响应**：PDF 解析和 AI 分析可能耗时，需使用异步任务队列 (Celery) 并在前端提供基于 SSE (Server-Sent Events) 的实时进度打字机推送。
3. **技术栈与大模型**：
   - **大模型**：优先使用支持超长上下文的 **DeepSeek** 大模型（已在环境变量中配置）。
   - **核心数据库**：升级采用 **PostgreSQL**，利用 `pgvector` 插件实现 RAG（检索增强生成）所需的向量存储，不再使用 SQLite。
   - **对象存储**：采用自建在 Docker 上的 **MinIO** 存储 PDF 原件、解析图表及生成的标书文档。
   - **核心后端框架**：FastAPI (Python)。
4. **交互体验**：界面需设计美观、专业且易用，契合 B 端效率工具的标准。

# 验收标准
1. 能成功上传并解析 10MB 以上的 PDF 招标文件，系统不崩溃。
2. 资质匹配功能必须能够清楚地输出三个等级的匹配结果，且有理有据。
3. 必须能准确提取并醒目提示文件中的“废标项/一票否决项”。
4. 必须能正确解析出“评标办法”或打分规则表，并形成可视化提示。
5. 生成的大纲结构必须与招标文件要求的“投标文件格式”严格一致。
6. 能够准确提取采购清单，并能关联参考价格库计算出总成本预估。
7. 能够识别出明显的免责声明或违约金等风险项。
8. 导出的 Word/Excel 文档必须可以正常使用 Office 或 WPS 打开并编辑。
