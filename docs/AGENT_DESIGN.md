# 智能投标系统 - Agent 架构设计指南

本文档总结了智能投标辅助系统（bidding_sys）的 Agent（智能体）架构设计原则，以及在项目中如何设计多智能体协同与调度。

## 1. 基础认知：Agent 核心模块
一个成熟的 Agent 区别于普通对话机器人的关键在于其架构包含四大模块：
1. **核心大脑 (Persona/Profile)**：设定角色、边界和目标驱动力（System Prompt）。
2. **记忆系统 (Memory)**：分为短期记忆（当前对话上下文）和长期记忆（借助向量数据库，使用 RAG 技术检索历史项目或公司资质）。
3. **规划与思考 (Planning)**：任务拆解 (Task Decomposition)、ReAct (思考-行动-观察) 以及自我纠错能力。
4. **工具与行动 (Tools)**：赋予大模型改变世界的能力，如搜索网页、查询数据库、调用外部 API 等。

## 2. 项目专属架构：专家委员会模式 (Multi-Agent)

针对处理超长、非结构化的招标文件（PDF/Word），并且要求极高准确率的业务痛点，单体大模型无法胜任。系统应采用 **“专家委员会”** 模式，将复杂任务拆分给四个各司其职的数字员工（作为底层工具 Tools 存在）：

*   🕵️ **拆解智能体 (Extractor Agent)**
    *   **定位**：数据清洗员。
    *   **职责**：负责调用 OCR/PDF 解析工具，将长文档切片（Chunking），把段落和表格存入向量数据库，为其他 Agent 提供上下文。不负责深度逻辑推理。
*   👮 **合规/排雷智能体 (Compliance Agent)**
    *   **定位**：风控法务。
    *   **职责**：带着极强的目的性去向量库（RAG）中检索带 `*` 号、“废标”、“否决投标”字样的条款。严格按 Prompt 提取风险条款及依据，并落库。
*   🧑‍🏫 **策略计分智能体 (Strategy Agent)**
    *   **定位**：商务与投标专家。
    *   **职责**：提取“评标办法”和“打分表”。对比公司基础业务数据（资质库），生成红绿灯匹配度分析，提示哪些能拿满分，哪些存在短板。
*   👨‍💻 **标书撰写智能体 (Writer Agent)**
    *   **定位**：文案秘书。
    *   **职责**：在流程末端，读取状态黑板（State）中各位专家的结论和企业知识库，自动起草商务/技术偏离表及标书初稿。

## 3. 控制流设计：混合路由架构 (Hybrid Routing)

在多智能体系统中，是否需要一个上位的 **Supervisor Agent（调度/主控智能体）** 来指挥其他子 Agent？
**答案是：需要，但必须分场景使用（双轨制）。**

### 3.1 快车道：静态工作流（无需 Supervisor）
*   **场景**：用户点击前端界面的特定功能按钮（如“一键排雷”、“生成打分表”）。
*   **调度方式**：代码硬编码的工作流（如 LangGraph 的有向无环图 DAG）。直接由 FastAPI 路由触发 `Compliance Agent` 或按序执行几个 Agent。
*   **优势**：速度极快，100% 稳定，绕过大模型意图识别的开销，杜绝了“大模型幻觉导致流程中断”的风险。

### 3.2 慢车道：动态路由（必须使用 Supervisor）
*   **场景**：全局 AI 对话助理窗口。用户输入模糊指令（如：“帮我看看这个标书废标风险大不大？如果有风险，怎么规避？”）。
*   **调度方式**：**Router / Tool Calling** 模式。
    *   引入 `Supervisor Agent`，将底层的 `Compliance`, `Strategy` 等专家 Agent 封装成它的 Tools。
    *   Supervisor 负责理解自然语言（Reasoning），自行判断需要调用哪些专家、以及调用的先后顺序。
    *   子专家执行完毕后将结果返回给 Supervisor，最终由其汇总并用自然语言答复用户。
*   **优势**：赋予系统极高的智能化和交互灵活性，能够处理代码 IF-ELSE 无法穷举的用户请求。

## 4. 落地技术选型建议

在 Python + FastAPI 的后端生态下，强烈推荐使用 **LangGraph** 进行 Agent 编排：
1. **状态机支持 (Blackboard)**：LangGraph 为“多 Agent 共享状态流转”设计，所有 Agent 共用一个 State（黑板），实现无缝的信息传递。
2. **人机交互 (Human-in-the-loop)**：支持在关键节点（如表格解析完毕后）暂停，等待前端用户确认后再继续后续 Agent 的工作。
3. **后台异步化**：高度兼容 Celery 等异步任务队列，长耗时的 Agent 工作流可在后台运行，并通过 SSE 实时推送进度（打字机效果）给前端。

**核心开发思路：**
优先花精力打磨底层的原子化“专家 Agent”，确保它们作为独立的 Python 函数/工具足够稳定。然后，即可在固定流程中直接调用它们，或在对话场景下将其喂给 Supervisor Agent 进行动态调度。
