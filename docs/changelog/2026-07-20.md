# 2026-07-20 改动记录 (Changelog)

## 核心优化：动态意图路由 (Dynamic Intent Routing) 与智能全局搜索

今日针对大模型提取标书信息时发生的“幻觉”（随意编造数据）和“视野狭窄”（因为限制章节导致漏提核心条款）问题，进行了系统性重构与修复：

### 1. 引入 RoutingService (意图导航引擎)
- **新增模块**：在 `app/services/routing_service.py` 中实现了 `RoutingService`。该模块利用 LLM 在执行具体提取前，先分析用户检索意图与标书大纲 (TOC)，动态返回该意图最可能存在的“目标章节列表”。
- **目的**：避免大模型在毫不相关的章节中因局部关键词匹配而产生幻觉（例如从商务条款中错误提取技术参数）。

### 2. 路由引擎的“智能全局/局部判定”升级 (Hotfix V2)
- **现象**：第一版的强制路由导致了“过度裁剪”。例如，财务金额和资质门槛跨越了“第一章”和“第四章”，但路由引擎只指定了“第四章”，导致大模型漏掉了 118 万的预算和核心资质要求。
- **解决方案**：在 `RoutingDecision` 数据结构中新增 `is_global_search` 布尔字段。
- **机制重写**：
  - 更新了底层的提示词。当大模型判定用户的查询（如财务、资质）是**跨章节的分散知识**时，强制设定 `is_global_search = True`，让 RAG 服务进行全文地毯式搜索，防止漏提。
  - 对于**高度聚焦的知识**（如评标办法、评分表），则设定 `is_global_search = False`，精准锁定目标章节（如第五章），防止被其他不相关的表格干扰。

### 3. 工具层 (metadata_tools.py) 的调度逻辑与靶向词优化
- **双路联动**：在所有的 `extract_xxx` 提取工具（资质、财务、商务、技术）中接入 `routing_service`。如果工具未收到明确章节限定，则自动启动智能路由决策机制。
- **修复评标提取的路由偏差**：
  - 在 `extract_evaluation_info` 评标罚则提取工具中，将传入路由引擎的诱导词**强制剥离了“罚则”部分**（仅传入“评标办法 评分标准 商务分 技术分 价格分 权重”）。
  - **效果**：确保路由引擎坚定地将其识别为“局部知识”，精准锁定评标章节（第五章），从而完美提取出原先丢失的 100 分权重分布和打分树。
- **靶向关键词扩充**：
  - 资质合规：增加了 `"特定资格要求 营业执照 失信被执行"`，确保与第一章的常用术语高度契合。
  - 商务时限：增加了 `"交付时间"`，配合“工期”进行双重打靶。

### 4. 异常处理与兜底机制
- 优化了 `RoutingService` 的容错逻辑。如果文档本身没有有效大纲 (TOC 缺失)，或者大模型发生异常，系统会自动将 `is_global_search` 置为 `True` 触发降级全量搜索，确保后续系统不会崩溃且有数据返回。
- 整个流程通过在 `_extract_and_format` 中打印丰富的 `emit_agent_log` 实时日志，方便在前端追踪大模型的路由决策过程。

## 进阶优化：绝对稳定性、防幻觉与界面统一

在解决完提取逻辑漏提后，进一步从大模型采样随机性和业务规则层面进行了加固：

### 5. 前端展示标签的语义统一
- **修改位置**：`frontend/src/components/dashboard/EvaluationCard.tsx`。
- **优化内容**：将原本的标题与卡片小标题“售后及硬性服务红线”精简、统称为 **“售后服务”**，使其更加贴合常见的招投标话语体系，且不破坏后端的 Pydantic 数据契约（后端仍保持 `hard_service_requirements` 键名以兼容历史数据）。

### 6. 售后服务提取 Prompt 的精准微调与排雷
- **扩展捕获范围**：在 `app/services/metadata/evaluation_service.py` 中，修改了传入大模型的系统提示词，强制要求其不仅关注罚则，还要去捕获“售后服务”、“维护保养”、“培训服务”等具体内容。
- **完美平衡“防幻觉”与“防误杀”**：
  - **初版雷区**：由于严格要求“如果不涉及具体数值强制返回 null”，导致大模型将整个字典置为 `null`，从而触发了 Pydantic 严格字典校验的 `ValidationError`（幸运的是被 Master Agent 的自我纠错机制自动重试拦截修复）。
  - **最终解法**：
    1. 将 `EvaluationSchema` 中的 `hard_service_requirements` 类型放宽至 `Optional[dict[str, Any]]`，允许合法空值。
    2. 将防幻觉的红线规则软化为：**提取的内容必须具备具体数值边界或“明确动作指向”的硬性要求（如“必须提供原厂质保函”）**。如果仅是“态度好”等假大空废话则强制丢弃，从而既拦截了废话，又保住了没有数字的干货要求。

## 资质中心模块 (Qualification Center) - 全新开发与架构升级

本次开发完成了企业资质中心的核心全栈闭环，并成功解决了扫描版 PDF 多资质解析的业务痛点。

### ✨ 新增特性 (Features)
- **后端模型与 CRUD**：创建了 `CompanyQualification` 的数据库存取层与 Pydantic 校验模型（支持多租户 `X-Tenant-ID`）。
- **后端 API**：新增了 `GET /api/v1/qualifications/`（列表）、`POST /upload`（上传识别）、`PUT /{id}`（更新）以及 `DELETE /{id}`（删除）的 RESTful 接口。
- **前端页面与导航**：在侧边栏新增了“资质中心”入口，并开发了 `/qualifications` 页面（瀑布流/网格卡片布局展示，支持各种到期状态颜色的自动切换：🟢有效 / 🟠即将过期 / 🔴已过期）。
- **动效弹窗与骨架屏**：设计并开发了高颜值、带毛玻璃动效的拖拽上传组件 `QualificationUploadModal`，配有直观的 MinerU OCR 识别加载动画（机器人闪烁）。

### 🚀 核心架构重构与优化 (Enhancements)
- **多模态 OCR 识别流水线**：摒弃了基础的 PyMuPDF 纯文本提取，采用强力级联方案 `MinerUParser` -> `Markdown` -> `LLMService`，完美支持含有图片、印章的扫描版资质文件的识别。
- **单文件多资质提取 (V4升级)**：
  - 更新了 LLM 提取大纲 Schema (`QualificationExtractionResult` > `List[QualificationExtractedData]`)，允许单次从一份文档中抠出所有堆叠在一起的不同资质。
  - 后端实现了一个文件产生多条数据库记录的批量关联。
  - 前端改造了上传核对界面：从原来的单个输入框升级为支持动态生成的列表型输入组，允许用户一次核对多个解析出的资质并执行批量保存。
- **本地代理网络阻断修复**：修复了在开发环境中由于挂载 127.0.0.1 代理导致上传 Aliyun OSS (MinerU 云端) 发生 `SSLError` (EOF) 的问题，修改 `mineru_parser.py` 发起 requests 时显式旁路本地代理 (`proxies={"http": None, "https": None}`)。

## 多模态大模型 (VLM) 视觉直传解析 (新增)
- **智能分流系统**：修改 `CompanyQualificationService`，对于 `.pdf` 仍保留使用 MinerU 深度解析；对于图片类文件跳过 OCR，直接走新加入的 **VLM 视觉解析链路**，避免了长篇复杂表格引发的提取缺漏。
- **双模型引擎随意切换**：加入了“阿里云通义千问模型”与“本地 Minimax 模型”双引擎。只需在 `.env` 及 `config.py` 里更改 `VLM_PROVIDER` 的值（`ali` 或 `local`），就能一键切换线上或线下模型调用。
- **输出标签剥离器**：在 JSON 解析前加入了强大的正则表达式剥除逻辑，解决了模型习惯性用 ` ```json ` 格式包装结果导致 `JSONDecodeError` 的问题。
- 新增独立的 `test_vlm_api.py` 测试脚本用于独立校验各类模型表现基准，并移除了线上环境的原始 JSON 磁盘日志留存逻辑。

### 🌟 资质解析深度修复与优化 (Hotfixes)
- **CRUD 数据落库修复**：修复了在 `backend/app/db/crud/qualification.py` 中 `create_qualification` 方法实例化 `CompanyQualification` 时遗漏 `company_name` 字段的问题，确保大模型提取出的公司名称能正确写入数据库并在前端展示。
- **VLM Prompt 遵循防幻觉黄金准则**：全面优化了 `company_qualification_service.py` 中的看图提取 Prompt。摒弃了原本带有“甲建筑公司”的写实型示例（Few-Shot），改用“XXX企业”、“YYYY-MM-DD”等抽象占位符，彻底防止大模型因锚定效应照抄示例数据产生幻觉，同时确保模型能正确学习并执行“将一张大证书拆分为多个具体资质小项”的复杂逻辑。
- **Python f-string 转义修复**：修复了因在 Python 格式化字符串 (`f-string`) 中直接嵌套单层 JSON 大括号 `{}` 而引发的 `Invalid format specifier` 后端崩溃错误，统一修正为 `{{}}` 双层括号安全转义。
- **编码灾难修复**：修复了在 Windows 下因终端写入导致的 `.env` 中文注释变乱码并引发 `uvicorn` 核心启动报 `UnicodeDecodeError` 崩溃的问题，重新以标准的 UTF-8 编码重写了环境配置文件。
- **VLM 规则级深度优化（防漏提）**：针对《建筑业企业资质证书》拆分时大模型出现的“只给第一项写日期，后续子项日期留空”以及“级别漏读”问题，在 Prompt 中新增了强硬的**信息继承规则**与**等级分离规则**，强制要求其将日期与公司名全量拷贝到每一个拆分后的资质中，并死盯后缀的等级词汇，确保提取 100% 完整。

### 🌟 资质展示与交互升级 (UI & UX)
- **批量管理加强**：在 `QualificationCenter.tsx` 的批量管理模式下新增了“全选/取消全选”功能。
- **视图重构 (Grid -> Table)**：将原本的卡片网格布局重构为紧凑、清晰的数据表格 (Table) 视图，更符合后台管理系统的操作直觉，同时完美兼容状态标识与多选控制。
- **图片预览方案升级**：移除了表格内冗余的“OCR文本”查看按钮；开发了**系统内沉浸式图片预览弹窗 (Modal Preview)**。现在点击图片原文件会直接在当前页面居中弹出毛玻璃视窗预览，而对于 PDF 则保留浏览器原生新标签页阅读。
- **本地静态资源路径打通**：
  - 后端：在 `main.py` 中通过 `StaticFiles` 挂载了 `backend/uploads` 本地目录，使 FastAPI 服务器具备分发用户上传图片的能力。同时重构了 `CompanyQualificationService` 的基础路径，使其将文件保存到 `backend/uploads` 而不是项目根目录。
  - 前端：在组件中自动拼接了 `API_BASE_URL`，解决了前后端端口分离导致的本地开发环境下图片链接 `404` 破损问题。
- **大模型长列表提取截断修复 (防偷懒机制)**：针对提取 10 条以上资质项时大模型容易在末尾“截断丢失”的问题，在 `company_qualification_service.py` 里的 VLM 与文本模型提示词中双双增设了 **“数量核解强制规则”**，严禁擅自停止或省略末尾条目，强制保证提取总数与原文件完全一致。
- **到期时间精准提取规则 (防错读机制)**：针对体系认证证书上印有多个日期（如发证日期、监督审核日期、有效期至）导致大模型误读的问题（例如错把 2025 年的年审日期当作 2027 年的最终作废日期），新增了强约束指令，强制模型只提取带有“有效期至”或范围截止日期的最终时间。
- **公司主体鉴别规则 (防张冠李戴)**：针对含有多个主体（如被认证公司、发证机构/认证机构盖章）的证书，新增了精确提取指令，强迫大模型只能提取真正的“被认证方/持证方”，严禁误将诸如“北京大恒航宇质量认证中心”之类的发证机关当作所属公司提取。
- **延迟落库重构 (防脏数据污染)**：重构了 AI 智能解析的交互流。在此之前，文件一经上传，后端便直接把提取的数据强制写进数据库；现在调整为“内存态渲染 -> 用户手动修正/删除不合理项 -> 点击【确认保存】按钮”的逻辑链条，仅当用户确认无误时，才会批量提交 POST 请求进行数据库落库操作，避免了提取失败或幻觉产生的数据污染数据库。
- **Pydantic Schema 幻觉诱导词修复**：排查发现 `QualificationExtractedData` 的 `expiry_date` 字段描述中存在“如果是长期有效，可为空或者推延一个较长日期”的**错误诱导提示**，导致大模型在遇到无日期的证书时会自行编造假日期。现已将该 Schema 的描述修复为“如果无明确日期请直接填 null，严禁自行编造或推延日期”，彻底根除了无中生有的日期幻觉。
- **Prompt 示例长度填充幻觉修复**：发现之前的 Prompt 中的 `【重要示例】` 强行提供了一个包含 3 条资质的 JSON 数组模板。这导致视觉大模型在识别出图片只有 2 项时，会触发强烈的“模仿行为”，无中生有地编造出第 3 条资质（甚至抄袭旧示例里的假日期 2025-04-28）来凑齐 3 个。现已将示例精简为 1 条极简格式，彻底打断了大模型“凑字数”的毛病。
- **资质名称与业务范围混淆修复**：针对大模型错误地将“业务范围/许可范围/认证范围”（如：电力工程的施工）当作“资质名称”提取的问题，在提示词中增设了**“资质名称鉴别规则 (防业务范围误导)”**，强制要求模型只抓取顶部醒目的大字标题（如《质量管理体系认证证书》），严禁把范围描述当作证书名。
