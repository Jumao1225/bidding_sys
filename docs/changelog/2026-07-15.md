# Bidding Sys 日常改动日志 - 2026-07-15

## 长文档切分策略重构（语义聚类与表格上下文保护）

### 问题背景
在处理超长章节进行自适应切片时，旧版 `_adaptive_split_chapter` 会将「表格」完全独立提取剥离成单一 Chunk，导致表格丢失了前后的描述文本和二级标题等核心上下文（Context）；同时，普通的文本切分仅根据字数阈值一刀切，容易把同一个二级标题（`##`）下的内容强行割裂到不同 Chunk 中，影响 RAG 检索的语义完整度。

### 修复方案：基于二级标题(H2)的语义聚类 + 表格上下文弹性容忍
在 `extractor_service._adaptive_split_chapter` 中实现全新切片流程：

1. **H2 语义聚类 (Semantic Grouping)**：首先将超长大章按 Markdown 的 `\n## ` (二级标题) 切分为多个逻辑段落（H2 Section）。
2. **整段合并优先**：遍历每个 H2 Section，若累积字数加上当前 Section 长度不超过 `MAX_CHUNK_SIZE`（1200字），则将整个二级标题下的内容紧密合并到一个 Chunk，**确保二级段落的完整性**。
3. **表格及其上下文保护**：对于被迫拆分的极长二级段落，在遇到不可分割的`table`块时：
   - 只要当前 Chunk 的前置文本较短（`< 300字`，通常为表格的标题和引导语），即便加上巨大表格会超过 1200 字，也会**容忍该 Chunk 超限**。
   - 这确保了表格与其紧邻的前后文说明始终“捆绑”在一个片段中，再也不会出现表格被孤立提取或被撕裂的情况。
4. **统一注入标头**：为拆分出的每一个子 Chunk 的最开头统一补全其所属的**大章标头**（如`【第四章 项目需求】`），防止深层分块后大章语境丢失。

### 改动文件
- `app/services/extractor_service.py`: 彻底重构 `_adaptive_split_chapter` 切片方法，实现二级标题语义绑定与表格弹性缓冲保护。

## Metadata Service 架构重构与 Master Agent 工具化升级 (自主提炼与双擎落盘)

### 问题背景

   - 暴露 `search_bidding_document` 作为兜底通用 RAG 工具，用于处理特殊、碎片的查询。
4. **Master Agent 升级 (LangGraph ReAct)**：
   - 重构了 `app/agents/supervisor.py`，向 `create_react_agent` 中一次性注入 6 个工具。
   - 更新系统 Prompt，赋权 Master Agent 像真实人类专家一样“自主分析、连续调用专属工具”去抽丝剥茧，最后汇总输出。专项工具负责底层精准提取与数据库写入，Master Agent 负责宏观协同。

## 解决数据乱码与 RAG 语义稀释，引入“精细检索”与“双重刹车自纠错”

### 问题背景
1. **中文转义乱码**：存入数据库的 JSON 数据，其中文被自动转义为 Unicode（如 `\u5b89\u88c5`），导致后续查看和读取时可读性极差。
2. **RAG 语义稀释 (Semantic Dilution)**：在进行诸如“最高限价”这样精确数字的提取时，之前的“整章召回”策略把包含目标的一整章文本全塞给了模型，加之检索词（如`最高限价 预算`）过长导致匹配度下降，模型常常因为注意力分散而提取出 `null`。
3. **Agent 自我纠错陷入死循环**：若要求 Agent 遇到 `null` 时自我重试，如果没有明确的业务判断和熔断边界，当标书中“明确说明不需要”或“确实没写”时，Agent 会不断换词重试，浪费资源并卡死流程。

### 修复方案：确保可读性、RAG 双轨制与自检双重刹车
1. **解决 Unicode 转义**：
   - 优化 `app/db/session.py`，配置 SQLAlchemy 的 JSON 序列化器 `custom_json_serializer`，强制指定 `ensure_ascii=False`，保证落库的中文原汁原味。
2. **RAG 精细检索“外科手术刀”模式**：
   - 为 `rag_service.search_bidding_document` 新增 `context_mode` 与 `query_mode`。
   - **分词多路检索 (`split`)**：底层自动将搜索词拆散去重，分别 Embedding 后聚合召回，避免长难词组导致的失配。
   - **±1 局部滑窗 (`window`)**：废弃整章召回，只精确抓取命中切片及其上下各一句切片，彻底剔除噪音干扰，大模型命中率飙升。
3. **Agent 强制纠错与“双重刹车”机制**：
   - 修改 `metadata_tools.py` 的提取工具底层，全部默认强制启用精细模式（`window` + `split`）。
   - **业务刹车（免死金牌）**：修改 `base.py` 中底层提取模型的 Prompt。当原文明确写了“无需提供”或“待定”时，不再返回 `null`，而是返回 `明确无要求` 或 `待定` 的占位符。
   - **物理刹车（强制熔断）**：在 `supervisor.py`（Master Agent 大脑）的系统指令中下达铁律——如果返回了特定占位符，**绝对禁止重试**；如果真没搜到（返回 `null`），每个工具**最多只能主动重试 2 次**，超过后必须接受现实推进流程。

### 改动文件
- `app/db/session.py`: 增加 `custom_json_serializer` 防止中文转义。
- `app/services/rag_service.py`: 新增多路分词 (`split`) 与局部滑窗 (`window`) 逻辑。
- `app/agents/tools/metadata_tools.py`: 各提取工具硬性挂载精细模式。

## 全链路 Unicode 转义彻查与修复

### 问题背景
在排查数据库及前端展示时，发现部分中文字符串被序列化成了 `\uXXXX` 形式的 Unicode 转义序列（如 `\u4f60\u662f\u8d44\u6df1\u7684`），根本原因是 **Python 标准库 `json.dumps` 的默认行为是 `ensure_ascii=True`**，会把所有非 ASCII 字符转义。对此进行了全项目的系统性扫描与修复。

### 扫描范围与结论

| 文件 | 是否存在问题 | 说明 |
|---|---|---|
| `app/db/session.py` | ✅ 已修复（历史记录） | `custom_json_serializer` 已配置 `ensure_ascii=False`，所有 SQLAlchemy JSONB 列落盘均安全 |
| `app/worker/tasks.py` - `publish_progress` | ❌ **存在问题** | Redis Pub/Sub 推送缺少 `ensure_ascii=False` |
| `app/worker/tasks.py` - `emit_agent_log` | ❌ **存在问题** | Agent 思考日志推送缺少 `ensure_ascii=False` |
| `app/agents/tools/metadata_tools.py` | ❌ **存在问题** | Pydantic v2 `model_dump_json()` 默认 `ensure_ascii=True` |
| `app/services/llm_service.py` | ✅ 无问题 | `schema_json = json.dumps(..., ensure_ascii=False)` 已正确 |
| `app/agents/supervisor.py` | ✅ 无问题 | `logger.info(json.dumps(..., ensure_ascii=False))` 仅用于日志，不落库 |
| `app/agents/nodes/cost_agent.py` | ✅ 无问题 | `json.dumps(price_book, ensure_ascii=False)` 已正确 |

### 修复方案

1. **`tasks.py` - Redis 消息推送**：
   - `publish_progress` 与 `emit_agent_log` 两处 `redis_client.publish(...)` 调用均补加 `ensure_ascii=False`，确保 SSE 前端接收到的中文内容不再乱码。

2. **`metadata_tools.py` - Pydantic 序列化**：
   - 原代码 `metadata_obj.model_dump_json(indent=2)` 使用 Pydantic v2 原生序列化，该方法**不接受 `ensure_ascii` 参数**，且默认转义非 ASCII 字符。
   - 修复方案：改用 `json.dumps(metadata_obj.model_dump(), indent=2, ensure_ascii=False)`，先将 Pydantic 对象解包为 Python 字典，再通过标准库安全序列化，同时也兼容了 Pydantic v1/v2 的写法差异。

### 改动文件
- `app/worker/tasks.py`: `publish_progress` 与 `emit_agent_log` 两处 `json.dumps` 均补加 `ensure_ascii=False`。
- `app/agents/tools/metadata_tools.py`: 将 `model_dump_json(indent=2)` 替换为 `json.dumps(model_dump(), ensure_ascii=False)`，根除 Pydantic v2 的隐式 ASCII 转义。
