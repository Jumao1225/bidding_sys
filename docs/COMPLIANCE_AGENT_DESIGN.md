# 合规与排雷智能体 (Compliance Agent) 深入设计文档

本文档专门针对 `Compliance Agent`（排雷智能体）在实际复杂业务场景下（如：标书中并未明确使用 `*` 号标注实质性要求）的应对策略和底层逻辑设计进行详细说明。

## 1. 业务痛点分析

在规范的招标文件中，通常会使用 `*` 或 `▲` 明确标识出“实质性要求”（即乙方不满足就会导致废标或扣分的条款）。
但在大量真实的业务场景中，很多限制性条款和“暗坑”并无明显标识。如果 Agent 仅仅依赖正则匹配或寻找特殊符号，将会导致漏判，给投标带来致命风险。

## 2. 核心排雷策略 (无星号场景)

为了应对上述痛点，`Compliance Agent` 需要从简单的“符号查找器”升级为“具备法律风控意识的审查员”。具体通过以下三个维度来实现：

### 2.1 语义与强限制词检索 (Hybrid Search)
在向量检索（RAG）阶段，不应盲目让大模型阅读全文，而是先通过**双路召回（BM25 关键词 + 向量相似度）**捞取高危段落。
需要建立高危词汇字典，强制命中包含以下特征的文本块：
*   **绝对性/强制性词汇**：必须、应当、严禁、不得、一票否决、否则视为无效。
*   **高压/惩罚性词汇**：废标、否决投标、取消投标资格、没收投标保证金、违约金、赔偿、终止合同。

### 2.2 大模型语义审查与 Prompt 设定 (Prompt Engineering)
利用大模型的深度阅读和逻辑推理能力，通过强大的 System Prompt 赋予其“风控专家”的视角。

**关注维度的 Prompt 设计要素：**
*   **资质类暗坑**：审查是否有不合理的前置条件，例如“要求提供原件备查”、“要求特定年份的社保流水”。
*   **财务类暗坑**：识别霸王条款，例如不合理的付款节点（“背靠背付款”）、远超行业标准的违约金或质保金。
*   **技术类暗坑**：识别具有排他性的技术壁垒，例如“系统必须与某某老系统（竞争对手产品）无缝对接”、“指定某单一品牌的硬件”。

### 2.3 行业与企业规则库外挂 (RAG 增强)
Agent 的判断不仅基于当前标书的内容，还应该结合外部规则库进行双向印证。
*   **建立《常见废标项》知识库**：将《政府采购法》中的法定废标条款，以及公司历史上“踩过坑”的教训整理成专用的向量库。
*   **双向比对逻辑**：Agent 拿到标书后，不仅在标书内寻找风险点，同时会拿着规则库里的“避坑指南”（如：没有法定代表人签字必废）去检查标书，看标书内是否有极其隐蔽但致命的要求。

## 3. 落地伪代码逻辑示例

```python
# 1. 强力词汇预检索
keywords = ["废标", "必须", "违约金", "取消资格", "没收"]
high_risk_chunks = vector_db.hybrid_search(document_id, keywords=keywords, top_k=20)

# 2. 规则库召回
rule_base = vector_db.get_company_risk_rules()

# 3. 大模型深度推理 (Tool Calling)
prompt = f"""
你是一名拥有20年经验的评标与法务风控专家。
目前的标书中可能隐藏了不带 '*' 号的实质性要求或财务陷阱。
请仔细阅读以下【高危段落】以及公司的【历史避坑指南】，找出所有潜在风险，并给出判断依据。

高危段落：{high_risk_chunks}
历史避坑指南：{rule_base}

返回格式要求：严格遵循 RiskItem 格式输出 JSON 数组，包含风险类型、原文、严重程度、你的判定依据。
"""

risk_report = llm.generate_structured_output(prompt, schema=RiskItemSchema)
```

通过这种多维度的结合，`Compliance Agent` 能够最大限度地规避“无星号”标书带来的业务风险，真正成为投标风控的护城河。
